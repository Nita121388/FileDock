# 2026-03-10

## P15 snapshot retention policy buckets

- Extended snapshot pruning beyond `keep-last` / `keep-days` so FileDock can now keep one newest snapshot per recent UTC day (`keep-daily`) and per recent ISO week (`keep-weekly`).
- Updated the shared prune request schema plus CLI flags, then taught the server retention selector to union all keep rules per device group before deleting old snapshot metadata/manifests.
- Added focused server unit tests covering the new day/week buckets and the union behavior between multiple keep rules.
- Refreshed retention/API/roadmap/script docs so the new policy options are discoverable from both docs and helper scripts.

## P16 agent/daemon scheduling (M2 plan)

- Drafted a concrete M2 proposal for agent scheduling: keep the existing long-running daemon mode as default, and add an opt-in scheduled-runner mode that uses OS-native schedulers to run a new `agent run-once` command periodically.
- Added `docs/agent-scheduling-m2.md` describing goals, CLI surface changes (`agent run-once`, `agent install --mode`), per-platform wiring (systemd timers / launchd StartInterval / Windows Task Scheduler), and acceptance/verification checklists.
- Updated `docs/roadmap.md` to list scheduling M2 as a near-term optional improvement.

## P17 agent/daemon scheduling (M2 implementation)

- Added `filedock agent run-once --config <path>`: runs exactly one snapshot (plus optional start/end heartbeats) and exits.
- Extended `filedock agent install` with `--mode daemon|scheduled` (default `daemon`):
  - `daemon`: keeps a long-running agent alive (cadence is owned by `interval_secs` in the profile TOML).
  - `scheduled`: installs an OS-native scheduler definition that periodically runs `agent run-once`.
- Implemented per-platform wiring (best-effort):
  - Linux: systemd user timer + oneshot unit; status reports the timer enabled/active state; uninstall removes both timer + service.
  - macOS: LaunchAgents `StartInterval` runner (no `KeepAlive`).
  - Windows: Task Scheduler periodic task (whole-minute interval derived from `interval_secs`).
- Updated agent docs (`docs/agent.md`) and added focused unit tests for the new service/timer render templates.

## P18 desktop onboarding: scheduling mode switch

- Extended the desktop guided agent setup flow to let users choose install mode:
  - `daemon` (default): keep a long-running background process alive.
  - `scheduled`: wire OS-native schedulers to periodically run `agent run-once`.
- Updated Tauri bridge + frontend types/i18n so `Preview install` and `Install and start` can pass `--mode scheduled`.
- Kept backward compatibility: only passes `--mode` when scheduled is selected.

## P19 desktop restore: conflict policy

- Extended the desktop restore-to-folder backend request to support a `conflict_policy` flag: `overwrite` / `skip` / `rename`.
- Implemented per-file conflict handling in the Tauri restore worker:
  - `overwrite`: current behavior (clobbers existing paths).
  - `skip`: treats existing files as done (keeps progress consistent).
  - `rename`: picks a unique destination name like `file (restore abcd1234).ext`.
- Updated the Device Browser header controls to include a conflict policy selector next to restore concurrency (persisted in local storage; defaults to `rename`).
- Refreshed `docs/quickstart.md` + `docs/roadmap.md` so the restore workflow is discoverable.

## P20 ignore rules: default excludes + visibility

- Seeded agent profiles with a small default exclude list when users do not provide `--exclude` / `--ignore-file` (keeps common giant trees like `.git` and `node_modules` out of backups by default).
- Extended the agent init summary (CLI JSON) so desktop onboarding can surface the effective ignore/exclude settings.
- Improved `filedock status` visibility by supporting `--exclude` / `--ignore-file` plus an `--include-ignored` mode for debugging ignore behavior.
- Pruned ignored directories during folder walks so both `push-folder` and `status` avoid scanning huge ignored trees.

## P21 ignore rules: default excludes opt-out

- Added `filedock agent init --no-default-excludes` to disable seeding the default exclude list when users want a true "include everything" backup.
- Extended the desktop onboarding UI with a toggle for the recommended default excludes (still defaults on).
- Updated docs and added unit tests covering the default-excludes application logic.

## P22 ignore rules: optional root .gitignore support

- Added an opt-in `--respect-gitignore` flag to `push-folder`, `push-folder-loop`, `status`, and `agent init` (and stored it in the generated agent profile TOML).
- Implemented gitignore-style matching by parsing the root `.gitignore` file (current scope: root-only; nested `.gitignore` files are not yet supported).
- Kept backward compatibility: `.filedockignore` continues to use glob-per-line semantics; `.gitignore` rules are applied in addition when enabled.
- Added unit tests covering gitignore directory rules and `!` whitelist behavior, plus updated docs to document the new flag and current scope.
