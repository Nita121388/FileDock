# 2026-02-01

## Phase 7 (Browse/Restore MVP)

- Added snapshot browse endpoints:
  - `GET /v1/snapshots/:snapshot_id/tree?path=`
  - `GET /v1/snapshots/:snapshot_id/file?path=...`
- Added CLI commands:
  - `filedock tree` (list directory)
  - `filedock pull-file` (download one file)
- Updated API/build docs.

## Phase 8 (Multi-Chunk + Pull-Folder)

- Manifest supports multi-chunk files (backward compatible with legacy `chunk_hash`).
- Client uploads files in fixed-size chunks (MVP: 4MiB chunks).
- Server reconstructs files from multiple chunks for `GET /file`.
- Added CLI command: `filedock pull-folder` (restore entire snapshot).

## Phase 9 (Snapshot Index + List)

- Server stores snapshot metadata under `snapshots/<id>.json`.
- Added endpoint `GET /v1/snapshots` to list snapshots.
- Added CLI command: `filedock snapshots`.

## Phase 10 (Integrity + Fewer Requests)

- Server validates chunk content hash on upload (`PUT /v1/chunks/:hash`).
- Client batches `chunks/presence` for `push-folder` (reduces network round-trips).

## Phase 11 (Usability/Robustness)

- Server streams file downloads (avoids buffering whole files in memory).
- Client batches `chunks/presence` into smaller requests (safety for huge folders).
- Client retries chunk uploads with backoff + jitter.
- Added basic unit tests for protocol path/hash validation and disk storage key safety.

## Phase 12 (Concurrency + Progress)

- Added `--concurrency` to `push-folder` and `pull-folder`.
- `push-folder` uploads files in parallel and prints periodic progress.
- `pull-folder` downloads files in parallel and prints periodic progress.
- Added download retries with backoff + jitter.

## Phase 13 (Performance)

- `push-folder` no longer re-reads files that have no missing chunks (uploads are skipped, but manifest is still written).

## Phase 14 (Exclude Rules)

- Added `--exclude` glob patterns to `push-folder` to skip files during scan/upload.

## Phase 15 (Basic Auth - MVP)

- Server supports optional static token auth:
  - If `FILEDOCK_TOKEN` is set, all endpoints except `/health` require header `X-FileDock-Token`.
- CLI reads `FILEDOCK_TOKEN` and automatically sends the header on all requests.

## Phase 16 (TUI - Browse MVP)

- Added `apps/tui` (`filedock-tui`): split-pane terminal UI to browse:
  - snapshots list
  - directory tree within a snapshot (navigate into dirs / go back)
- TUI reads `FILEDOCK_TOKEN` and sends `X-FileDock-Token` automatically (same as CLI).

## Phase 17 (Desktop UI Shell - Tabs + Split Layout)

- Added `apps/desktop` (Tauri + Vite + React) UI shell:
  - Tabbed workspaces (single window, multiple tabs)
  - Split-pane layout tree (row/col splits) with draggable gutters to resize
  - Per-tab layout persistence (local storage)
- Added basic pane types (mock data for now): device browser, transfer queue, notes.

## Phase 18 (Desktop Docking - Drag to Split)

- Desktop panes support drag-and-drop docking:
  - Drag a pane titlebar onto another pane to reveal drop zones.
  - Drop on left/right/top/bottom to auto-split and dock the pane.
  - Drop on center to merge into target pane as tabs.

## Phase 19 (Desktop Pane Tabs)

- Leaf panes now support multiple internal tabs (so one pane can show multiple devices/snapshots/views).
- Center-drop docking merges panes into the target pane as tabs (Wave-like behavior).

## Phase 20 (Desktop: Live Snapshots + Tree)

- Device Browser pane now talks to the running FileDock server:
  - lists snapshots (`GET /v1/snapshots`) and groups by `device_name`
  - browses tree (`GET /v1/snapshots/:id/tree?path=`)
  - downloads a file via browser download (`GET /v1/snapshots/:id/file?path=...`)
- Added connection inputs in top bar:
  - server base URL
  - token (optional, sent as `X-FileDock-Token`)

## Phase 21 (Desktop: Contextual Pane Tabs)

- Pane tabs now carry their own view state (per tab):
  - for Device Browser: selected device, snapshot id, and current path
- Switching pane tabs restores the corresponding snapshot/path without losing other tabs' context.

## Phase 22 (Desktop: Transfer Queue MVP)

- Added a simple transfer queue (download jobs) stored locally.
- From Device Browser file list:
  - `+Q` queues a file download into the Transfer Queue pane.
  - Transfer Queue pane can execute the download per job (browser download).

## Phase 23 (Desktop: Drag File -> Queue)

- Files in Device Browser are draggable.
- Dropping a file onto the Transfer Queue pane enqueues a download job.

## Phase 24 (Server: Device Registry MVP)

- Implemented minimal device registry endpoints:
  - `POST /v1/auth/device/register` returns `device_id` + `device_token` (stored server-side)
  - `GET /v1/devices` lists registered devices

## Phase 25 (Desktop: Device Registry UI)

- Desktop Device Browser now integrates the device registry:
  - fetches `GET /v1/devices` (shows device OS when available)
  - provides a small register form (device name + os) calling `POST /v1/auth/device/register`

## Phase 26 (Device Token Auth - MVP)

- Server auth middleware accepts device tokens (when `FILEDOCK_TOKEN` is configured):
  - `X-FileDock-Device-Id` + `X-FileDock-Device-Token`
  - Note: `/v1/auth/device/register` still requires the server token when `FILEDOCK_TOKEN` is set.
- CLI supports env vars: `FILEDOCK_DEVICE_ID` and `FILEDOCK_DEVICE_TOKEN`.
- Desktop stores and sends device auth headers after registration.

## Phase 27 (Desktop: Per-Tab Connection Profiles - MVP)

- Device Browser pane tabs can override connection settings (per tab):
  - server url
  - server token
  - device id + device token
- This enables browsing multiple servers/devices within one window at the same time.

## Phase 28 (Snapshots: Link to Device ID)

- Snapshot create request supports optional `device_id` (from device registry).
- Snapshot metadata persists `device_id` (so UI can group/identify devices reliably).
- CLI `push-folder` includes `device_id` automatically when `FILEDOCK_DEVICE_ID` is set.

## Phase 29 (Desktop: Cross-Server Copy MVP)

- Drag a file from one Device Browser pane and drop it onto another Device Browser pane:
  - enqueues a `copy_file` transfer job (source server -> destination server).
- Transfer Queue can run `copy_file` jobs:
  - downloads bytes from source snapshot file endpoint
  - chunks + BLAKE3 hashes in the desktop app
  - uploads missing chunks to destination
  - creates a new snapshot on destination and writes a one-file manifest

## Phase 30 (Desktop: Copy Target Path UX)

- Drag-drop copy now respects the destination pane's current directory:
  - dropping onto a destination Device Browser pane copies into its current `path/` (instead of preserving full source path).

## Phase 31 (Desktop: Local File Upload MVP)

- Device Browser tree now supports uploading a local file into the selected device/path:
  - `UL` button opens a file picker and uploads the file (chunked + dedup) to the current server.
  - Dropping an OS file onto the tree uploads it (same behavior).
  - Uploads create a new one-file snapshot on the destination device (MVP behavior).

## Phase 32 (Desktop: Queue UX + Streaming Upload)

- Transfer Queue has quick actions:
  - Run queued / retry failed / run all
  - clear done
- Transfer jobs now have a `running` status (UI pill).
- Local upload no longer buffers the entire file in memory:
  - hashes file by reading slices
  - uploads missing chunks by reading slices on demand

## Phase 33 (Desktop: Transfer Progress + Retries)

- Transfer Queue shows per-job progress (when available):
  - phase text (downloading / uploading / finalizing)
  - percent bar for downloads and cross-device copy
- Desktop transfers have basic retry with backoff + jitter for flaky networks.

## Phase 34 (Desktop: Queue Concurrency + Pause)

- Transfer Queue supports:
  - configurable concurrency (1-8, persisted locally)
  - pause/resume scheduling (running jobs finish; new jobs won't start while paused)

## Phase 35 (Desktop: Cancel Running + Auto-Run)

- Transfer Queue supports:
  - cancel a running job (best-effort via aborting in-flight fetches)
  - auto-run queued jobs (toggle, persisted locally)

## Phase 36 (Desktop: Bandwidth Limit - MVP)

- Transfer Queue supports a simple bandwidth limit input (MB/s, 0 = unlimited).
- Desktop client applies best-effort throttling to:
  - downloads (stream read backpressure)
  - cross-server copy uploads (delay between chunk PUTs)

## Phase 37 (Desktop: Directory Drag-Copy - MVP)

- Device Browser tree supports dragging directories between device panes.
- Dropping a directory onto a destination Device Browser pane enqueues a `copy_folder` job:
  - enumerates files via repeated `tree` calls (client-side)
  - creates one destination snapshot with a multi-file manifest (MVP)
  - supports resume via persisted `nextIndex` + `manifestEntries`

## Phase 38 (Desktop: Multi-Select + Batch Drag-Copy - MVP)

- Device Browser tree supports selecting multiple items (checkbox UI; shift-range and ctrl/cmd toggle).
- Dragging a selected item drags the whole selection (current directory view) onto another device pane:
  - files become `copy_file` jobs
  - directories become `copy_folder` jobs

## Phase 39 (Desktop: Copy-Folder Resume - MVP)

- `copy_folder` jobs are resume-friendly without large local storage usage:
  - destination manifest is read from server to detect already-copied paths
  - manifest is written to server after each file (MVP reliability)

## Phase 40 (Desktop: Copy Into Existing Snapshot - MVP)

- Drag-copy now supports a "base snapshot" on the destination pane:
  - when the destination Device Browser has a selected snapshot, copy jobs will base their manifest on that snapshot
  - a new snapshot is created on the destination (copy-on-write), then the updated manifest is written
- Copy-folder now supports conflict policy (skip/overwrite/rename) when writing into the destination manifest (MVP).

## Phase 41 (Desktop: Conflict Policy UI - MVP)

- Transfer Queue exposes conflict policy selection per copy job (overwrite/skip/rename).

## Phase 42 (Desktop: Chunk-Level Copy - Perf MVP)

- Copy jobs now prefer chunk-level transfers when source manifest includes chunk lists:
  - `copy_file`: fetch missing chunks from source server `/v1/chunks/:hash` and upload to destination (no full-file download)
  - `copy_folder`: same optimization per file when possible
- Major benefit: if destination already has the chunks, copy becomes mostly "manifest writes" (dedup hit).

## Phase 43 (Desktop: Cache Chunk Presence - Perf)

- For large `copy_folder` runs, the desktop now caches destination chunk presence results within the job:
  - reduces repeated `chunks/presence` calls across many files that share chunks
  - still uses batching (1000 hashes/request) and respects job cancellation (AbortSignal)
- `copy_file` also reuses the same presence cache within a job (helps when falling back between code paths).

## Phase 44 (Desktop: Parallel Chunk Copy - Perf)

- Chunk-level copy now transfers missing chunks with a small bounded concurrency (currently 4):
  - speeds up `copy_file` when it can use source chunk lists
  - speeds up `copy_folder` for chunk-level files
- Progress updates are now driven by bytes transferred (missing-bytes total), even when copying chunks in parallel.

## Phase 45 (Desktop: Parallel Files in Copy-Folder - Perf)

- `copy_folder` now copies multiple files concurrently (bounded concurrency, currently 4 max / at least 2):
  - per-file data transfer runs in parallel
  - destination manifest updates and `putManifest` writes are serialized to keep correctness + crash-resume behavior
- Resume logic now tracks a contiguous `nextIndex` even with out-of-order completion (avoids re-copying already-finished files).

## Phase 46 (Desktop: Copy-Folder Concurrency Setting)

- Added a queue setting to control per-job `copy_folder` file concurrency (1..8, default 4).
- Desktop `copy_folder` now reads this setting from `filedock.desktop.queue.v1` (same store as queue concurrency + bandwidth limit).

## Phase 47 (Desktop: Cross-Job Chunk Presence Cache - Perf)

- Added a shared, per-destination-server chunk presence cache inside the desktop app:
  - caches "have" results with an LRU cap (reduces repeated `chunks/presence` checks across multiple jobs)
  - caches "missing" results only briefly (30s) to avoid long-lived false-missing
- Uploading a chunk now updates the shared cache immediately (missing -> have).

## Phase 48 (Desktop: Deduplicate Concurrent Chunk Uploads)

- When multiple concurrent tasks try to upload the same chunk hash to the same destination server, the desktop now de-duplicates the `PUT /v1/chunks/:hash` calls:
  - first uploader performs the upload
  - followers await the in-flight upload promise
- Reduces wasted bandwidth when `copy_folder` runs with file-level concurrency and many files share chunks.

## Phase 49 (Desktop: Safer Cache Keys + Faster Cancel)

- Shared destination caches (presence + in-flight chunk uploads) are now keyed by full connection identity (server URL + auth fields), not just `serverBaseUrl`.
- Waiting on an in-flight chunk upload now respects cancellation (followers can abort without hanging).

## Phase 50 (Desktop: Better Copy-Folder Progress)

- `copy_folder` progress messages now include an overall `[done/total]` file prefix while copying.
- Internal completion tracking now maintains a dedicated `filesDoneCount` counter (works with out-of-order completion under file-level concurrency).
